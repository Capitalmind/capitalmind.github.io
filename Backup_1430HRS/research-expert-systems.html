The Core Challenge

How do we build logic-based systems for domains that require zero margin for error—such as game rules, engineering standards, or legal compliance—while maintaining transparency, auditability, and natural language accessibility?

What Are Expert Systems & Ontologies?

Expert Systems: Transparent AI

Expert systems represent a fundamental approach to artificial intelligence that prioritizes transparency, explainability, and deterministic reasoning. Unlike neural networks that operate as "black boxes," expert systems provide complete audit trails from input through reasoning to conclusion.

Black Box AI

  • Probabilistic outputs (85% confident)
  • No explanation of reasoning
  • Cannot guarantee consistency
  • Difficult to audit or validate
  • Good for pattern recognition

Expert Systems

  • Deterministic outputs (100% confident)
  • Complete reasoning chains
  • Guaranteed consistency
  • Full audit trails
  • Perfect for rule enforcement

Ontologies: The Knowledge Foundation

An ontology is a formal representation of knowledge that defines entities, their properties, and relationships within a specific domain. Think of it as a blueprint that captures how things connect and interact in a structured, machine-readable format.

Why Games Make Perfect Examples

Games provide ideal testing grounds for expert systems because they have clearly defined rules, measurable outcomes, and zero tolerance for ambiguity. If we can build a system that perfectly enforces complex game rules, we can apply the same techniques to legal frameworks, engineering standards, or any domain requiring deterministic logic.

Case Study: Flutter Stock Exchange Game

To demonstrate ontology development, we'll use the Flutter Stock Exchange Game—a complex strategy game with intricate rules perfect for showing how human-readable rules translate into machine-enforceable logic.

Understanding the Game Rules

Before building any ontology, we must thoroughly understand the domain. Here are the core Flutter Stock Exchange rules that we'll convert into formal logic:

Flutter Stock Exchange - Core Rules

Game Objective

Each player starts with £300. Win by reaching £600 total value (cash + shares at market prices) through strategic trading and dividend collection.

Key Components
  • Six Companies: Fairfit Furniture, Atomic Airways, Easygoing Engineering, Diggers Diamond Mines, Bluebottle Brewery, Captivating Cosmetics
  • Two Pegs per Company: Parent peg (market price) and Traveller peg (moves up board)
  • Share Certificates: Each represents 100 shares, starting value £100 PAR
  • Market News Cards: Random events affecting company values
Trading Rules (Rule 8)
  • Players may only buy/sell shares immediately before throwing dice
  • Shares bought/sold in multiples of 100 only
  • Purchase price = market price + £5 brokerage fee
  • Sale price = current market price (no fees)
  • All transactions through broker only
Dividend Payouts (Rule 11)
Traveller Peg Position Dividend per 100 Shares Parent Peg Movement
Top of board (200+) £20 (20%) Up 2 spaces
15% position (180-189) £15 Up 1 space
10% position (170-179) £10 Up 1 space
5% position (160-169) £5 Up 1 space
Market News (130) None No movement
Below Market News None Down 1 space
Special Events
  • Bonus Shares (Rule 12): When parent peg reaches £200, issue 1:1 bonus shares, reset price to £100
  • Bankruptcy (Rule 13): Parent peg reaches bottom = company ceases trading, shares worthless
  • Slump (Rule 10): Traveller peg at "SLUMP" position drops back 6 spaces automatically
  • Market News (Rule 9): Landing on "Market News" triggers random card effects
Why These Rules Are Perfect for Ontology Design

The Flutter game contains all the complexity patterns found in real-world systems: entity relationships (players, companies, shares), state management (prices, positions), action validation (trading rules), conditional logic (dividend calculations), and event handling (special positions). Every rule must be enforced with zero errors—exactly like legal or engineering systems.

Ontology Development Process

Step-by-Step Methodology

Building effective ontologies requires systematic analysis. Here's the proven process for converting complex rule systems into machine-readable knowledge graphs:

Phase 1: Domain Analysis

Before writing any code, we must understand the domain deeply. This means identifying all entities, relationships, and rules that govern the system.

1. Identify Core Entities
  • GameState: Complete game instance
  • Player: Participants with money and shares
  • Company: Tradeable entities with stock prices
  • Broker: Manages share transactions
  • MarketCard: Random events affecting gameplay
2. Map Relationships
  • Player → owns → Shares
  • Company → has → StockPrice
  • GameState → contains → Players
  • Broker → manages → Certificates
  • MarketCard → affects → Company

Expert System Implementation

The ontology provides the structure, but we need a rule engine to process logic. Here's how we implement the expert system:

# Expert System for Game Rule Enforcement # Demonstrates zero-error validation using ontology data class GameRuleExpert: """Expert system for enforcing complex game rules with zero errors""" def __init__(self, ontology_path): # Load game ontology (Flutter Stock Exchange in this case) self.ontology = self.load_game_ontology(ontology_path) self.game_state = self.ontology['gameInstance'] # Extract rule sets from ontology self.trading_rules = self.extract_trading_rules() self.movement_rules = self.extract_movement_rules() self.win_conditions = self.extract_win_conditions() # Audit trail for complete transparency self.decision_log = [] def validate_share_purchase(self, params): """ Demonstrate complete rule chain validation Uses ontology data to ensure zero-error enforcement """ player = params['player'] company_id = params['company'] quantity = params['quantity'] rule_chain = [] # Rule 1: Company must be active (from ontology) company = self.get_company_by_id(company_id) if not company['isActive']['@value']: rule_chain.append("VIOLATION: Company must be active for trading") return (False, "Company not active", rule_chain) rule_chain.append(f"✓ Company {company_id} is active") # Rule 2: Sufficient certificates available (from broker) broker = self.ontology['broker'] available = self.get_available_shares(broker, company_id) if available < quantity: rule_chain.append(f"VIOLATION: Only {available} shares available, {quantity} requested") return (False, "Insufficient shares", rule_chain) rule_chain.append(f"✓ {available} shares available, {quantity} requested") # Rule 3: Player has sufficient cash (price + brokerage) share_price = company['parentPosition']['@value'] brokerage_fee = self.game_state['brokerageFee']['@value'] total_cost = (share_price * quantity) + brokerage_fee player_cash = player['currentCash']['@value'] if player_cash < total_cost: rule_chain.append(f"VIOLATION: Need £{total_cost}, player has £{player_cash}") return (False, "Insufficient funds", rule_chain) rule_chain.append(f"✓ Player has £{player_cash}, cost £{total_cost}") rule_chain.append("✓ ALL VALIDATION RULES PASSED - Transaction allowed") return (True, "Purchase approved", rule_chain)
